## 三、答辩讲稿(详细版)

### 开场(同学 A,1 分钟)

> 各位老师好,我们小组的课程设计项目叫「城市光影 CityGlow」,
> 是一款基于光污染监测的星空寻回地图。
>
> 城市的灯光让 80% 的城市居民看不到银河,我们的项目聚合气象数据、
> 光污染等级和天文事件,告诉用户「今夜是否值得仰望星空」,
> 并用一张可交互的暗夜地图,帮助用户寻回最后一片可以观星的暗夜之地。
>
> 我负责后端架构和 JDK 21 新特性的应用,[同学 B] 负责前端架构和视觉交互设计。
> 下面我先介绍后端,再由 [同学 B] 介绍前端。

---

### 后端讲解(同学 A,4-5 分钟)

#### 1. 技术栈与架构(1 分钟)

> 后端用 Spring Boot 3.2.5 + JDK 21,数据库是 MySQL 8,
> 用 Spring Data JPA 做 ORM,Spring Security + JWT 做用户认证,
> Caffeine 做本地缓存。
>
> 架构是经典的前后端分离:前端 Vue3 通过 REST API 调后端,
> 后端再调 Open-Meteo 和 NASA APOD 两个第三方 API。
> 所有 API 用统一的 `ApiResponse<T>` 包装,返回 `{code, message, data}` 结构,
> 前端 axios 拦截器统一解包,简化错误处理。

#### 2. JDK 21 虚拟线程(1 分钟,重点讲)

> JDK 21 的虚拟线程是我重点应用的新特性。
>
> 在 `ForecastService` 里,我需要并行调用 Open-Meteo 的多个端点(当前天气 + 日出日落),
> 传统做法是用 `CompletableFuture` 或线程池,代码复杂且资源开销大。
>
> 我用 `Executors.newVirtualThreadPerTaskExecutor()` 创建虚拟线程执行器,
> 每个任务跑在一个虚拟线程上,虚拟线程由 JVM 调度,不占用 OS 线程,
> 内存开销只有几 KB,可以轻松创建上千个。
>
> 配合 Spring Boot 的 `spring.threads.virtual.enabled=true`,
> Tomcat 的请求处理线程也自动切换到虚拟线程,整体吞吐量显著提升。

**答辩可能提问**:
- 老师:虚拟线程和传统线程池有什么区别?
- 答:虚拟线程由 JVM 管理,不绑定 OS 线程,在 I/O 阻塞时会自动让出载体线程,
  内存开销从 MB 级降到 KB 级。传统线程池适合 CPU 密集型,虚拟线程适合 I/O 密集型。

#### 3. 结构化并发(1 分钟,重点讲)

> 另一个 JDK 21 特性是结构化并发(preview 特性,需 `--enable-preview`)。
>
> 在 `PostcardService` 生成明信片时,需要并行做三件事:
> 压缩图片、查询月相、查询 Bortle 等级。这三件事任一失败就应该全部取消,
> 避免部分成功导致脏数据。
>
> 我用 `StructuredTaskScope.ShutdownOnFailure` 创建作用域,
> 在里面 `fork` 三个子任务,然后 `join` + `throwIfFailed`。
> 如果任一子任务抛异常,作用域自动关闭其他子任务,保证「要么全成功,要么全失败」。
>
> 这比 `CompletableFuture.allOf` + 手动异常处理简洁得多,代码可读性也更好。

**答辩可能提问**:
- 老师:为什么不用 CompletableFuture?
- 答:CompletableFuture 的异常处理很麻烦,`allOf` 不会在任一失败时取消其他,
  需要手动写取消逻辑。结构化并发天然支持「失败即关闭」,语义更清晰。

#### 4. JWT 认证与缓存(1 分钟)

> 用户认证用 Spring Security + JWT 0.12.x。
>
> 注册时密码用 BCrypt 哈希存储,登录成功后签发 JWT Token,
> 前端存在 localStorage,后续请求带 `Authorization: Bearer <token>` 头。
>
> 我写了 `JwtAuthenticationFilter`,在 SecurityFilterChain 里拦截需要鉴权的请求,
> 校验 Token 并设置 `SecurityContext`。`SecurityConfig` 里精确配置了放行规则:
> 公开接口(`/api/v1/sky/**`、`/api/v1/astro/**` 等)`permitAll`,
> 收藏接口 `/api/v1/favorites/**` 必须 `authenticated`。
>
> 缓存用 Caffeine:NASA APOD 缓存 24 小时(避免重复调第三方),
> 观星预测缓存 30 分钟,星图视图缓存 1 小时。用 `@Qualifier` 注入对应的 Cache Bean。

**答辩可能提问**:
- 老师:JWT 和 Session 有什么区别?
- 答:JWT 是无状态的,服务端不存 Session,Token 自包含用户信息和签名,
  适合分布式和前后端分离架构。缺点是无法主动失效(除非维护黑名单)。

### 前端讲解(同学 B,4-5 分钟)

#### 1. 前端架构与 Leaflet 地图(1 分钟)

> 前端用 Vue 3 `<script setup>` + TypeScript + Vite,
> 状态管理用 Pinia,路由用 Vue Router 4。
>
> 首页是暗夜地图,用 Leaflet 1.9 + CartoDB Dark Matter 暗色瓦片,
> 叠加 Bortle 光污染等级覆盖层和天文事件标记(彩色圆点)。
>
> 点击地图任意位置,右侧滑出观星条件预测面板。
> 这个交互用 Vue 的 `<Transition>` + TailwindCSS 实现,
> 面板从右侧滑入,带毛玻璃背景和淡入动画。

#### 2. Canvas 星座投影(1.5 分钟,重点讲)

> 我最想展示的是星图连接器功能,这是用 Canvas 从零实现的星空可视化。
>
> 后端返回的星点数据是地平坐标(方位角 Az + 高度角 Alt),
> 我在前端做立体投影:天顶在 Canvas 中心,地平线在边缘,
> 公式是 `r = (90 - alt) / 90 * maxR`,`x = cx + r * sin(az)`,`y = cy - r * cos(az)`。
>
> Canvas 上渲染了 12 个主要星座的星点(大小按视星等)+ 连线(暗金 60% 透明)
> + 地平线环 + 16 方位刻度 + 四角铜版画装饰。
>
> 鼠标交互方面,悬停时检测点到星座中心的距离,命中则高亮该星座(连线变 100% + 辉光),
> 其他星座降到 30% 透明度。点击弹出神话故事卡,双栏展示希腊神话和中国古代星宿对照。

**答辩可能提问**:
- 老师:为什么用 Canvas 不用 SVG?
- 答:Canvas 适合大量元素(上百颗星)的实时重绘,性能优于 SVG(SVG 是 DOM 操作)。
  时间滑块拖动时星空要实时旋转,Canvas 重绘更快。

#### 3. 视觉设计与动画(1 分钟)

> 视觉方向是「17 世纪铜版画星图 × 东方水墨」,配色严格用项目自定义的深空色板:
> 深空黑 `#0a0e1a` 做背景,暗金 `#c5a572` 做强调,月光蓝 `#b0c4de` 做次强调。
>
> 字体用 Cormorant Garamond(衬线显示字体,用于标题和星座名)、
> Manrope(正文)、JetBrains Mono(坐标和时间)三套字体搭配。
>
> 动画细节:页面加载时 Canvas 整体淡入,时间滑块防抖 200ms 避免频繁调 API,
> 神话卡从中心 scale 展开弹出,月相用 Canvas 绘制带立体光影效果。

#### 4. 多语言与用户体验(1 分钟)

> 用 vue-i18n 9 实现中 / 英 / 日三语切换,所有文案抽到 `locales` 文件,
> 切换时导航、页面标题、月相名称、Bortle 描述全部实时更新,无需刷新。
>
> 用户体验细节:
> - 暗夜地图点击后预测面板自动滑出,不用额外跳转
> - 星图页面自动定位,失败回退北京坐标
> - 收藏点在星图页面下拉直接切换,不用手动输坐标
> - 移动端响应式,汉堡菜单 + 自适应网格

## 四、老师可能提问与参考答案

### 后端相关(同学 A 主答)

**Q1:为什么选 Spring Boot 3.2 而不是 2.x?**
> Spring Boot 3.x 要求 JDK 17+,能支持 JDK 21 的新特性(虚拟线程、Record、Pattern Matching)。
> 3.2 还原生支持虚拟线程(`spring.threads.virtual.enabled`),2.x 不支持。

**Q2:虚拟线程在什么场景下有优势?什么场景下反而更慢?**
> 优势场景:I/O 密集型(HTTP 调用、数据库查询、文件读写),因为虚拟线程在 I/O 阻塞时会自动让出载体线程。
> 劣势场景:CPU 密集型计算,虚拟线程没有优势,甚至因为调度开销略慢。这种情况用平台线程池更好。

**Q3:JWT Token 怎么主动失效?(比如用户改密码后)**
> 标准 JWT 是无状态的,无法主动失效。两种方案:
> 1. 维护一个 Token 黑名单(Redis),每次校验时查一下
> 2. 缩短 Token 有效期(如 1 小时),配合 Refresh Token 续期
> 本项目是课程设计,用短期 Token + 退出时前端清 localStorage 的简化方案。

**Q4:Caffeine 缓存和 Redis 缓存有什么区别?为什么选 Caffeine?**
> Caffeine 是本地(进程内)缓存,Redis 是分布式缓存。
> Caffeine 优势:零网络延迟,不需要额外部署,适合单机场景。
> Redis 优势:支持多实例共享,支持持久化,适合分布式部署。
> 本项目是单机部署,且缓存数据(APOD 图片 URL、预测结果)丢失影响不大,选 Caffeine 足够。

**Q5:Open-Meteo 为什么不用 OpenWeatherMap?**
> OpenWeatherMap 的 One Call API 3.0 需要付费订阅,免费 tier 调用会 401。
> Open-Meteo 是开源免费的,无需 API Key,无速率限制,数据质量也足够,
> 更适合课程设计场景。

### 前端相关(同学 B 主答)

**Q6:为什么用 Vue 不用 React?**
> Vue 3 的 `<script setup>` 语法更简洁,学习曲线平缓,适合课程设计快速开发。
> React 的 JSX 灵活性更强,但对两人小组来说 Vue 的单文件组件更易协作。

**Q7:Canvas 投影算法是你自己写的吗?怎么验证正确性?**
> 是我自己实现的。验证方法:北极星(Polaris)的赤纬是 89.26°,
> 从北京(纬度 39.9°)观测,北极星的高度角应该约等于纬度(39.9°)。
> 后端单元测试验证了这一点,计算结果是 39.3°,偏差 0.6°,在容差范围内。

**Q8:为什么用 TailwindCSS 不用普通 CSS?**
> TailwindCSS 的原子类让样式直接写在 HTML 里,不用来回切文件,
> 自定义色板配好后,全项目色彩一致性很容易维护。
> 缺点是 HTML 会变长,但对课程设计项目来说开发效率更重要。

**Q9:三语切换怎么实现的?后端也要改吗?**
> 用 vue-i18n 9,纯前端实现。所有文案抽到 `locales/zh.ts`、`en.ts`、`ja.ts` 三个文件,
> 切换时用 `i18n.global.locale.value = 'en'` 即可,不需要刷新页面,后端完全不用改。

**Q10:星座连线数据从哪来?**
> 我参考了 Stellarium(开源天文软件)的星座连线图,手动整理了 12 个 MVP 星座的连线数据,
> 每个星座 5-15 条线段,连接亮星的 HIP 编号对。数据存在后端 `constellations.json` 里。
